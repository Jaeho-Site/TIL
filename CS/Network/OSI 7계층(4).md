    ## 5. [종합] 1.4MB 파일을 전송할 때 일어나는 일
    
    지금까지 학술적으로 살펴본 OSI 7 Layer의 철학, TCP/IP의 실무 구조, 식별자(MAC, IP, Port)의 역할, 커널 소켓 메커니즘, 그리고 캡슐화와 단위(MTU, MSS) 분할의 개념을 모두 종합해 볼 차례다. 송신측 서버에서 수신측 클라이언트 PC로 **1.4MB (약 1,400,000 바이트)** 크기의 이미지 파일을 전송하는 구체적인 시나리오를 시간의 흐름에 따라 심층적으로 재구성해 보자. 이 통합 시나리오를 온전히 이해한다면 네트워크 인프라 전반의 거시적 설계와 미시적 데이터 처리 메커니즘을 완벽하게 관통할 수 있다.
    
    ### 송신측: 퍼즐 쪼개고 택배 포장하기 (Encapsulation)
    
    1. **소켓을 통한 파일 데이터의 진입 (L7 -> L4)**:
    서버 시스템의 파일 전송 애플리케이션(L7, 유저 모드)은 하드 디스크에서 1.4MB 크기의 이미지 파일을 읽어들여 자신의 애플리케이션 메모리 공간에 적재한다. 전송 준비를 마친 애플리케이션은 클라이언트와 이미 연결이 수립된 소켓(Socket)의 파일 디스크립터를 향해 `write()` 또는 `send()` 시스템 콜을 호출하여 1.4MB의 데이터를 들이붓는다. 이 순간 데이터는 권한의 경계를 넘어 유저 영역을 떠나 커널 영역 깊숙이 위치한 TCP 스택으로 진입하게 된다.
    2. **데이터의 정밀 분할 및 TCP 캡슐화 (L4 Segment 생성)**:
    데이터를 넘겨받은 커널의 TCP 스택은 1.4MB라는 용량이 네트워크 케이블이 감당하기엔 너무나도 거대하다는 사실을 인지한다. 따라서 사전에 연결 시 합의된 MSS 크기(1460 바이트)를 기준으로 데이터를 기계적으로 토막 내기 시작한다. 1,400,000 바이트를 1460 바이트로 나누면, 대략 958개에서 959개의 작은 조각들이 생성된다.
    TCP는 분할된 이 958개의 각 데이터 조각 맨 앞에 20바이트 분량의 TCP 헤더를 정성스럽게 덧붙인다. 헤더 내부에는 출발지 포트 번호(예: 웹 서버 80), 목적지 포트 번호가 기입되며, 가장 중요한 정보인 '순서 번호(Sequence Number)'가 각 조각마다 순차적으로 매겨진다. 수신측이 나중에 이 번호를 보고 뒤죽박죽 도착한 958개의 퍼즐 조각을 순서대로 맞출 수 있도록 배려한 것이다. 이로써 958개의 완전한 **세그먼트**가 탄생한다.
    3. **IP 캡슐화 및 최적 경로 라우팅 (L3 Packet 생성)**:
    생성된 958개의 세그먼트는 커널 내부에서 엘리베이터를 타듯 차례대로 하위 계층인 IP 스택으로 내려간다. IP 스택은 각 세그먼트 앞에 20바이트의 IP 헤더를 다시 한번 결합한다. 이 봉투 표면에는 출발지 서버의 고유 IP 주소와 최종 목적지인 클라이언트 PC의 IP 주소가 명확하게 적힌다. IP 프로토콜은 목적지 IP 주소의 서브넷 마스크(Subnet Mask)를 분석하여 해당 목적지가 내부망에 있는지 외부망에 있는지 판단하고, 외부망일 경우 라우팅 테이블을 조회해 이 패킷을 어느 라우터(게이트웨이)로 던져야 할지 다음 경로를 결정한다. 이 과정을 통해 네트워크 최대 한계인 1500바이트 크기를 꽉 채운 958개의 **패킷** 포장이 완료된다.
    4. **MAC 캡슐화 및 전기/광 신호 변환 (L2 Frame -> L1 Bit)**:
    소프트웨어 영역인 커널에서의 모든 처리를 마친 958개의 패킷은 메인보드의 버스를 타고 하드웨어인 랜카드(NIC)의 송신 큐(Queue)로 이동하여 차곡차곡 쌓인다. 랜카드 컨트롤러는 다음 목적지(대개는 첫 번째 라우터)의 물리적 주소를 ARP(Address Resolution Protocol)를 통해 확인한 후, 패킷 앞부분에 목적지 MAC 주소와 출발지 자신의 MAC 주소가 담긴 헤더를 씌우고, 뒷부분에는 에러 검출용 FCS 트레일러를 덧붙여 완전한 **프레임** 형태로 포장한다.
    최종적으로 랜카드는 이 프레임들을 0과 1의 아날로그 전기 신호(비트 스트림)로 실시간 변환하여, 랜선이라는 고속도로로 쉴 새 없이 쏘아 보낸다. 마치 958대의 택배 트럭이 일제히 물류 센터를 빠져나가 목적지를 향해 질주하는 순간과 같다. 패킷이 중간 라우터를 거칠 때마다 IP 패킷 상자는 그대로 유지되지만, 그 상자를 싣고 달리는 트럭(MAC 프레임)은 각 구간의 물리망 특성에 맞춰 계속해서 갈아타게 된다.
    
    ### 수신측: 택배 수령 후 퍼즐 조립하기 (Decapsulation과 Buffer)
    
    1. **신호 수신 및 프레임 무결성 검증 (L1 -> L2)**:
    수많은 라우터를 거친 신호가 마침내 수신측 클라이언트 컴퓨터의 랜카드에 도달한다. 랜카드는 케이블을 통해 밀려 들어오는 아날로그 전기 신호를 감지하고 이를 즉시 디지털 데이터로 조립하여 프레임 구조로 복원해 낸다. 가장 먼저 프레임 헤더를 열어 목적지 MAC 주소가 자신의 하드웨어 MAC 주소와 일치하는지 대조한다. 일치한다면 프레임 끝부분의 FCS(체크섬)를 계산하여 긴 여행 동안 전기적 노이즈에 의해 데이터가 1비트라도 훼손되지 않았는지 무결성을 엄격하게 검증한다. 검증을 무사히 통과하면 랜카드는 MAC 헤더와 트레일러를 벗겨내는 1차 역캡슐화를 수행하고, 하드웨어 인터럽트(Interrupt)를 발생시켜 남은 알맹이(패킷)를 운영체제 커널의 메모리 공간(IP 스택)으로 복사해 올려보낸다.
    2. **목적지 IP 확인 및 TCP 스택 전달 (L3 Decapsulation)**:
        
        커널의 IP 스택은 수령한 패킷의 IP 헤더를 분석한다. 앞선 비유처럼 L3의 "택배 기사(IP 프로토콜)"는 오직 봉투에 적힌 목적지 IP 주소만 기계적으로 확인하며, 상자 내부의 내용물이나 순서 따위는 전혀 궁금해하지 않는다. 목적지 IP가 자신의 컴퓨터 주소가 정확히 맞다면 IP 스택은 임무를 다한 IP 헤더를 제거하고, 알맹이인 세그먼트를 L4 전송 계층의 TCP 스택으로 인계한다.
        
    3. **순서에 맞춘 퍼즐 조립과 소켓 버퍼링 (L4 TCP Buffer)**:
    TCP 스택은 IP로부터 958개의 세그먼트를 연이어 수신하며 본격적인 검수 및 조립 작업에 돌입한다. 인터넷이라는 네트워크 환경은 극도로 불안정하기 때문에, 958개의 조각 중 일부가 혼잡한 라우터에서 유실되어 도착하지 않거나, 각각 다른 경로를 타고 와서 순서가 뒤죽박죽으로 섞인 채 도착할 확률이 매우 높다. TCP 프로토콜은 세그먼트 헤더에 적혀 있는 '순서 번호(Sequence Number)'를 꼼꼼히 대조하여, 뒤섞인 조각들을 원래의 1.4MB 파일 순서대로 완벽하게 재정렬한다. 만약 누락된 조각을 발견하면, 송신측 서버에게 해당 번호의 조각만 다시 보내달라고 요청(Retransmission)하여 애플리케이션 데이터의 신뢰성을 100% 보장한다.
    재조립이 완료된 순수한 원본 데이터 조각들은, 목적지 Port 번호와 매핑된 해당 애플리케이션의 소켓이 소유하고 있는 커널 내부의 **'수신 버퍼(Receive Buffer)'** 메모리 공간에 차곡차곡 쌓이며 주인이 찾아오기를 기다리게 된다.
    4. **애플리케이션의 데이터 Read (L4 -> L7)**:
    마지막으로, 유저 모드에서 블로킹(대기) 상태로 머물고 있던 애플리케이션(예: 크롬 웹 브라우저)이 커널을 향해 해당 소켓 디스크립터를 대상으로 `recv()` 혹은 `read()` 시스템 콜을 호출한다. 커널은 자신의 소켓 수신 버퍼에 안전하게 조립되어 쌓여있던 데이터를 유저 영역에 위치한 애플리케이션의 메모리 공간으로 고스란히 복사해 준다. 이로써 958개로 조각났던 1.4MB의 이미지 파일이 비로소 완전한 형태로 사용자 화면에 아름답게 렌더링되며 길고 길었던 통신의 여정이 마무리된다.
    
    ### 통신 속도를 결정하는 Window Size
    
    1.4MB 파일 전송 과정을 미시적으로 살펴보면서 네트워크 공학에 대한 한 가지 중대한 의문이 생길 수 있다. "네트워크 케이블의 대역폭만 충분하다면, 송신측 서버는 아무런 제약 없이 958개의 패킷을 한 번에 다 쏟아부어도 문제가 없는 것일까?" 결론부터 명확히 말하자면, 안정성을 최우선으로 하는 TCP 네트워크망에서는 절대 수신자의 상태를 무시하고 무작정 데이터를 전송하지 않는다.
    
    수신측 클라이언트 PC의 운영체제 커널에 할당된 TCP 소켓의 수신 버퍼 메모리 크기는 무한하지 않다. 물리적인 RAM의 한계로 인해 정해진 크기(예: 64KB)만을 할당받는다. 만약 송신측 서버 장비의 CPU 성능과 네트워크 회선이 압도적으로 뛰어나 데이터를 1기가비트의 속도로 쏟아붓고 있는데, 반대로 수신측 클라이언트 PC는 구형 모바일 기기라서 애플리케이션이 커널 버퍼에서 데이터를 읽어가는 속도(`recv()`)가 매우 느리다면 어떤 참사가 벌어질까?
    수신측 커널의 TCP 버퍼는 순식간에 들어오는 데이터 조각들로 한계치까지 가득 차버리게 된다(Buffer Overflow). 버퍼가 가득 찬 이후에 쉴 새 없이 도착하는 불쌍한 후속 패킷들은 커널이 저장할 공간이 없으므로 가차 없이 메모리에서 버려지게(Drop) 된다. 패킷이 버려지면 송신측은 응답을 받지 못해 데이터를 유실한 것으로 오판하고, 동일한 데이터를 계속해서 재전송(Retransmission)하려고 시도함으로써 가뜩이나 혼잡한 네트워크망의 과부하를 더욱 극심하게 가중시키는 치명적인 악순환이 발생한다.
    
    이러한 재앙을 원천적으로 차단하기 위해 TCP 프로토콜은 **'흐름 제어(Flow Control)'**라는 매우 강력하고 지능적인 동적 제어 메커니즘을 가동한다. 수신측은 자신이 데이터를 무사히 받았음을 알리기 위해 송신측으로 응답 패킷(ACK, Acknowledgement)을 반환할 때, 단순히 받았다는 사실만 알리는 것이 아니다. 응답 패킷의 TCP 헤더 안에는 "현재 내 커널의 TCP 수신 버퍼에 남아있는 빈 공간의 여유 크기"를 정확한 바이트 단위로 명시해서 함께 통보해주는데, 이 수신측의 남은 빈 공간 크기를 기술 용어로 **'윈도우 사이즈(Window Size)'**라고 부른다.
    
    송신측 TCP 스택은 수신측이 매번 알려주는 이 Window Size 값을 철저하게 모니터링하며, 이 여유 공간을 단 1바이트라도 초과해서 데이터를 보내지 않는다. 만약 애플리케이션의 처리 속도가 느려져 수신 버퍼가 꽉 차버리면, 수신측은 Window Size를 0으로 설정하여 응답을 보낸다. 이를 수신한 송신측은 즉시 데이터 송출을 일시 중단(Wait)하고 대기 모드로 들어간다. 이후 수신측 애플리케이션이 마침내 `read()` 함수를 호출하여 버퍼에서 데이터를 비워내고 여유 공간(Window Size)이 다시 늘어났다는 새로운 응답 패킷을 보내오면, 송신측은 딱 그 늘어난 공간의 크기만큼만 후속 세그먼트들의 송출을 재개한다.
    
    즉, 통신 데이터를 일방적으로 밀어내는 것이 아니라 보내는 쪽의 송출 속도를 받는 쪽의 소화 능력에 맞추어 밀리초 단위로 동적으로 조절하는 것이다. 이처럼 보이지 않는 곳에서 쉼 없이 작동하는 우아한 피드백 루프(Feedback Loop) 덕분에, 전 세계에 흩어진 수억 대의 기기들은 제각각 성능의 편차가 극심함에도 불구하고 데이터의 오버플로우나 손실 없이 매우 안정적이고 평화롭게 대용량 파일을 끊임없이 주고받을 수 있는 것이다.
