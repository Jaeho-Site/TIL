## 0. 한 줄 요약

> **내 강의자료·필기·논문을 올리면 그걸 근거로 답하는 학습 코파일럿.** 사용자가 "3주차 트랜잭션 격리수준 요약하고 예상문제 만들어줘"라고 하면, 에이전트가 **문서 색인 → 검색(RAG) → 근거 인용 → 요약/퀴즈 생성**을 단계적으로 수행하고, **모든 답변에 원문 출처를 하이라이트**한다. PDF 파싱·청킹·인덱싱 같은 **무거운 연산은 Web Worker에서 처리**해 메인스레드를 막지 않는다. 동네비서가 "외부 시세 검색"이라면, 스터디파일럿은 **사용자 문서 RAG + 클라이언트 성능 엔지니어링**을 전면에 둔다.

## 1. 왜 이 주제인가 — 기술적 의의

`기획요구사항.md`의 5대 목표를 충족하되, 동네비서가 비워둔 **CS 연결 성능 최적화(목표3)** 를 극대화하고 "브라우저 동작 원리·대용량" 영역을 정조준한다.

| 핵심 기술 영역 | 본 프로젝트의 대응 요소 |
|------|------|
| CS 기본기·브라우저 동작 원리·대용량·검색 | **Web Worker**·가상화·캐싱·문서 검색 인덱싱 |
| 대용량 트래픽·BFF·성능 | RAG 검색 파이프라인·긴 문서 렌더 최적화 |
| 인앱 웹뷰 + Native Bridge + AI 도구 활용 | 인앱 웹뷰 + 파일·카메라 스캔·음성 + 멀티스텝 Agent |
| AI 네이티브 제품 경험 | RAG 에이전트 자체가 "AI를 잘 전달한" 사례 |
| React, 상태관리, 브라우저 동작 원리 | Zustand 복합 상태 + 렌더링/메모리 최적화 |

> **결론**: 동네비서가 웹뷰+실시간을 강점으로 한다면, 스터디파일럿은 **성능 최적화 깊이**가 강점이다. Web Worker·직렬화 비용·가상화·메모리는 `기획요구사항.md` 목표3과 "브라우저 동작 원리"에 가장 깊게 닿아, **기술 설명거리의 밀도가 가장 높다**.

## 2. 서비스 개요

### 2.1 문제 정의
LLM 일반 챗봇은 **(1) 내 자료를 몰라 환각하고, (2) 출처가 없어 신뢰하기 어렵고, (3) 시험 대비용 정리/문제화가 수작업**이다. 또한 수백 페이지 문서를 웹에서 다루면 **파싱·렌더가 메인스레드를 막아** 입력이 끊긴다.

### 2.2 솔루션
앱 안의 **AI 에이전트**가 사용자의 문서를 인덱싱해 **출처 기반 답변(RAG)** 을 주고, 요약·플래시카드·예상문제까지 도구로 생성한다. 무거운 연산은 Web Worker로 분리한다.

- **색인 플로우(웹 워커)**: PDF/이미지 업로드 → 워커에서 텍스트 추출 → 청킹 → 검색 인덱스/임베딩 → 진행률 스트림
- **질의 플로우(에이전트 주도)**: 질문 → 관련 청크 검색(RAG) → 재정렬 → 근거 인용해 답변 → 출처 하이라이트
- **학습 플로우(에이전트 주도)**: "예상문제 만들어줘" → 요약 → 퀴즈/플래시카드 생성 → 오답 복습 스케줄

### 2.3 사용자 가치
- 내 자료 기반이라 **환각이 적고 출처로 검증**된다
- 요약·문제·복습까지 **에이전트가 자동 생성**
- 수백 페이지 문서도 **끊김 없이** 다룬다(워커 분리)

## 4. 기획요구사항(`기획요구사항.md`) 5대 목표 매핑

### 4.1 ✅ 목표1 — 복잡한 상태관리
- **Zustand 전역 스토어 분리**: `docStore`(문서·청크·색인 진행률), `agentRunStore`(RAG 파이프라인 상태머신), `chatStore`(대화·스트리밍 버퍼·출처 매핑), `quizStore`(문제·오답·복습 스케줄), `uiStore`
- **서버 상태 vs 클라이언트 상태 분리**: 서버 RAG 결과·대화 히스토리는 **TanStack Query**, 색인 진행률·로컬 인덱스·낙관적 입력은 Zustand
- **Agent(RAG) 실행 상태머신**: `idle → parsing → embedding/indexing → retrieving → reranking → synthesizing → done/error` 를 명시적 상태로 모델링하고 UI에 1:1 매핑
- **워커 ↔ 스토어 동기화**: Web Worker가 보내는 진행률/부분 결과 메시지를 받아 스토어에 reduce(트랜스퍼러블 객체로 직렬화 비용 최소화)
- **캐싱 전략**: 동일 질의·동일 청크 임베딩 재사용 키 설계, staleTime/gcTime 튜닝, 문서 해시 기반 색인 캐시
- **(선택)다중 사용자**: 스터디룸 공동 질의 시 질문·답변·읽음 동기화

### 4.2 ✅ 목표2 — WebView / 모바일 연동
- **인앱 웹뷰** 전제(Android/iOS WebView)
- **Native Bridge**: `window.AppBridge` 추상화로 파일/카메라/마이크/푸시/safe-area 호출
- **파일 업로드**: PDF·이미지 문서 입력
- **카메라**: 칠판·필기 **스캔** → OCR 입력
- **음성 입력**: 음성 질문 → STT → 에이전트 질의
- **Push Notification**: 에빙하우스 곡선 기반 **복습 알림**, 장시간 색인 완료 알림
- **Deep Link**: 알림 탭 → 해당 플래시카드·문서 위치로 직접 진입
- **웹뷰 특수 문제 해결 스토리**: 대용량 파일 업로드 메모리, 키보드 가림, 토큰 주입, 뒤로가기 제어 → **핵심 기술 포인트**

> 브릿지 없는 브라우저에서는 File API/Web Speech/Web Push 폴백으로 동작하게 추상화.

### 4.3 ✅ 목표3 — CS 연결 성능 최적화 (이 안의 최대 강점)
- **Web Worker**: PDF 파싱·텍스트 청킹·(클라이언트) 임베딩/유사도 계산·검색 인덱싱을 **메인스레드에서 분리** → 색인 중에도 입력 끊김 제거. **트랜스퍼러블 객체/SharedArrayBuffer**로 직렬화 비용 절감
- **가상화**: 긴 문서 뷰어(수백 페이지)·검색 결과·플래시카드 리스트를 `@tanstack/virtual`로 가상화 → DOM 노드 최소화, 메모리 절감
- **메모이제이션**: 메시지·출처 하이라이트 컴포넌트 `React.memo` + 안정적 key, 스트리밍 중 리렌더 범위 격리
- **코드 스플리팅 / Lazy Loading**: PDF 뷰어·퀴즈·통계 뷰 동적 import (무거운 PDF 라이브러리 지연 로드)
- **이미지/페이지 최적화**: 문서 페이지 lazy 렌더, 썸네일 분리, blur placeholder
- **네트워크 최적화**: 스트리밍(SSE) 청크, 질의 debounce/취소(AbortController), 색인 청크 업로드 병렬·재시도 백오프
- **측정→개선 루프**: **메인스레드 블로킹 시간(Long Task)**·FPS·메모리를 Performance API/React Profiler/web-vitals로 before/after 수치화 → 개선 서사 핵심

### 4.4 ✅ 목표4 — Agent 기반 서비스 (단순 API 호출 X)
- 에이전트가 **도구를 선택·순차/병렬 호출**하는 멀티스텝 RAG 구조
- 도구 세트: `indexDocument`(색인), `retrieveChunks`(검색), `rerank`(재정렬), `summarize`(요약), `generateQuiz`(문제생성), `citeSource`(출처 매핑)
- 질문 분석 → 검색 → (근거 부족 시 재검색 반복) → 합성 → 출처 인용의 **루프**
- (AI 서브 담당) 프론트는 이 RAG 과정을 **이벤트 스트림으로 받아 투명하게 반영**

### 4.5 ✅ 목표5 — AI UX / 에이전트 인터페이스
- **Chat UI + Streaming Response**: 토큰 단위 스트리밍
- **에이전트 진행 시각화**: "문서 색인 78% → 관련 청크 5개 검색 → 답변 합성 중" 타임라인/스텝퍼
- **Tool 실행 과정 표시**: 검색된 청크·점수를 접이식 카드로 노출(왜 이 답인지 근거)
- **출처 하이라이트(차별 포인트)**: 답변 문장 ↔ 원문 위치 **양방향 하이라이트/점프**
- **실시간 피드백**: 검색 중간 결과 점진 표시, 사용자가 출처 범위 좁히기 개입
- **긴 응답 최적화**: 가상화 + 점진 마크다운 렌더
- **세션/스레드 관리**: 문서·과목별 스레드 분리, 학습 히스토리 복구

## 6. 상태 아키텍처 설계 (핵심 차별화)

```
[서버 상태 — TanStack Query]          [클라이언트 상태 — Zustand]
 - RAG 검색/답변 결과 (캐시 키)         - docStore: 문서·청크·색인 진행률
 - 대화 히스토리                        - agentRunStore: RAG 파이프라인 상태머신
 - 퀴즈/복습 스케줄                     - chatStore: 입력·스트리밍 버퍼·출처 매핑
        │                                 - quizStore: 문제·오답·복습
        └──────── 동기화 경계 ─────────────┘
                    ▲
   [Web Worker 레이어 — 파싱/청킹/임베딩/인덱싱 → 진행률·부분결과 메시지 → 스토어 reduce]
```

**RAG 실행 상태머신**
```
idle ──질문──▶ retrieving ──청크──▶ reranking ──▶ synthesizing(stream)
   ▲                                                      │
   │                                                      ▼
 done ◀── 출처 매핑/하이라이트 ◀──────────── (근거 부족? 재검색 반복)
   └──────────── error/cancel(사용자 중단) ◀───────────────┘

[색인 경로] parsing → chunking → embedding/indexing → ready  (모두 Web Worker)
```
> "왜 워커인가 / 직렬화 비용 / 트랜스퍼러블 / Long Task 제거"가 핵심 기술 포인트.

## 7. 성능 최적화 — 측정 기반 개선 계획

| 영역 | 측정 지표 | 개선 기법 | 기대 서사 |
|------|----------|----------|----------|
| 문서 색인 | **메인스레드 블로킹(Long Task)** | Web Worker 분리 + 트랜스퍼러블 | "색인 중 입력 지연 N초 → 0" |
| 긴 문서 뷰어 | 스크롤 FPS, DOM 노드, 메모리 | 가상화 + 페이지 lazy | "수백 페이지 끊김·메모리 절감" |
| 채팅 스트리밍 | 리렌더 횟수 | 메모이제이션 + 렌더 격리 | "메시지당 리렌더 X→1" |
| 첫 진입 | LCP, TTI | SSR + 코드 스플리팅(PDF 지연) | "초기 로딩 N초 → M초" |
| 네트워크 | 요청 수/취소 | RAG 캐싱 + AbortController | "중복 질의 제거" |

- FE 보조: 문서 뷰어·퀴즈/복습 UI, 디자인 시스템
- BE: 에이전트 백엔드(LLM·임베딩·벡터검색), 색인 서버, 파일 스토리지
- (인원에 따라) 데이터/AI: 청킹·재정렬·문제생성 품질

## 9. 마일스톤 (제안, 약 12주)

1. **W1–2 설계**: 상태 아키텍처·워커 메시지 프로토콜·RAG 이벤트 프로토콜·브릿지 인터페이스 확정
2. **W3–4 색인 코어**: Web Worker 파싱·청킹·인덱싱 + 진행률 스트림
3. **W5–6 RAG 채팅**: 검색→합성 스트리밍 + 도구 시각화 + **출처 하이라이트**
4. **W7 문서 뷰어**: 긴 문서 가상화·검색결과·페이지 lazy
5. **W8 학습 기능**: 퀴즈/플래시카드 생성 + 오답 복습 스케줄
6. **W9 멀티모달/웹뷰**: 카메라 스캔·음성 질문·복습 푸시·딥링크
7. **W10 성능**: Long Task·메모리·FPS 측정→개선 지표 확보
8. **W11 품질/도전**: 테스트·오프라인 인덱스·(도전) 스터디룸 공동 질의
9. **W12 마감**: 배포, 데모, 프로젝트 회고 정리

## 10. 핵심 기술 도전 과제

- **성능 병목 측정·개선** → Web Worker로 Long Task 제거, 가상화 메모리 절감 before/after
- **브라우저 동작 원리** → 메인스레드/이벤트루프, 직렬화 비용, 트랜스퍼러블, Long Task
- **복잡한 비동기 상태 설계** → RAG 파이프라인 상태머신 + 워커↔스토어 동기화 + 서버/클라 분리
- **AI UX 전달** → RAG 진행·출처 하이라이트 시각화
- **웹뷰 환경 문제 해결** → 대용량 파일 메모리·브릿지 추상화·딥링크
- **AI 개발 도구(Claude Code/Cursor) 활용** → 개발 과정 기록

## 11. 리스크 & 대응

| 리스크 | 대응 |
|------|------|
| 클라 임베딩/RAG 범위 과다 | 코어는 **서버 RAG**, 클라 워커는 파싱·청킹·검색 인덱싱으로 한정 |
| 대용량 PDF 메모리 | 페이지 단위 스트리밍 파싱·가상화, 상한·청크 처리 |

| AI 비용/지연 | 임베딩/검색 결과 캐싱, 스트리밍으로 체감 지연 완화, 모델 분기 |

## 부록 A. 동네비서와의 관계
- **공통**: 인앱 웹뷰 + Native Bridge + 멀티스텝 Agent + Agent 진행 시각화 + Zustand/TanStack 분리
- **차별**: 외부 시세 검색 → **사용자 문서 RAG + 출처 인용**, 웹뷰+실시간 강점 → **Web Worker·가상화 성능 엔지니어링 강점**
- **선택 기준**: "성능 최적화/브라우저 동작 원리"를 가장 깊게 파고 기술 깊이를 극대화하려면 스터디파일럿.
