## 0. 한 줄 요약

> **내 차의 모든 걸 챙겨주는 모빌리티 비서.** 사용자가 계기판·정비명세서·보험증서를 **카메라로 찍으면 OCR**로 읽어 들이고, "다음 엔진오일 언제 갈아?", "지금 위치에서 가까운 충전소 경로"라고 물으면, 에이전트가 **정비 이력 분석·소모품 교체 예측·주변 인프라 검색·중고 시세 도구를 단계적으로 호출**한다. 가족이 함께 쓰는 차는 **운행/충전 상태를 실시간 공유**한다. 동네비서가 "중고 매물 등록/탐색"이라면, 마이카메이트는 **소유 차량 라이프사이클 관리 + OCR + 실시간 공유차 상태**를 다룬다.

---

## 1. 왜 이 주제인가 — 기술적 의의

`기획요구사항.md`의 5대 목표를 충족하되, 동네비서가 비워둔 **모빌리티·차량 데이터·정비 도메인** 을 정조준한다.

| 핵심 기술/도메인 영역 | 본 프로젝트의 대응 요소 |
|------|------|
| 모빌리티·차량 데이터·정비 도메인 | 차량 정비·소모품·리콜·시세 관리가 **서비스의 본질** |
| React, 상태관리(Redux/Zustand), 브라우저 동작 원리 | Zustand 복합 상태 + 렌더링 최적화 |
| 인앱 웹뷰 + Native Bridge + AI 도구 | 인앱 웹뷰 + **카메라 OCR**·위치·푸시·딥링크 |
| 실시간·다중 사용자 | **가족 공유차 실시간 상태 동기화** |
| React+TS, 지도·검색, 성능 최적화 | 주변 정비소/충전소 지도·검색, 리스트 가상화·이미지 최적화 |

> **결론**: 동네비서가 중고거래(웹뷰+거래)를 다룬다면, 마이카메이트는 **모빌리티·차량 라이프사이클**을 정면으로 다루면서 공통 기반(웹뷰+OCR, 지도·성능)까지 포섭한다. 카메라 OCR이 서비스의 본질이라 **웹뷰/Native 연동(목표2) 비중이 가장 크다**.

## 2. 서비스 개요

### 2.1 문제 정의
차량 소유자는 **(1) 정비·소모품·보험·리콜 시점을 놓치고, (2) 종이/사진으로 흩어진 기록을 관리 못 하며, (3) 가족 공유차는 누가 언제 쓰는지·충전 상태가 안 보인다.** 정비소·충전소 탐색과 중고 시세 파악도 번거롭다.

### 2.2 솔루션
앱 안의 **AI 에이전트**가 문서·계기판을 **OCR로 디지털화**하고, 주행거리·이력을 근거로 **교체 시점을 예측**하며 **주변 인프라 경로·중고 시세**까지 처리한다. 공유차는 실시간 상태를 동기화한다.

- **기록 플로우(웹뷰 OCR)**: 계기판/명세서/보험증서 촬영 → OCR로 주행거리·항목 추출 → 사용자 확인 → 차량 기록에 반영
- **정비 플로우(에이전트 주도)**: "엔진오일 언제 갈아?" → 이력·주행거리 분석 → 교체 주기 계산 → 주변 정비소 검색 → 예약 안내
- **인프라 플로우(에이전트 주도)**: "근처 충전소 경로" → 위치·충전소 검색 → 경로/이동시간 → 혼잡도
- **공유 플로우(다중 사용자)**: 가족 공유차의 운행 중/주차/충전 %·위치를 실시간 동기화

### 2.3 사용자 가치
- 흩어진 기록을 **사진 한 장으로** 디지털화
- 소모품·리콜을 **놓치지 않고** 예측 알림
- 가족 공유차의 상태를 **실시간으로** 한눈에

## 4. 기획요구사항(`기획요구사항.md`) 5대 목표 매핑

### 4.1 ✅ 목표1 — 복잡한 상태관리
- **Zustand 전역 스토어 분리**: `garageStore`(보유 차량·소모품 상태·예측), `agentRunStore`(실행 상태머신), `fleetStore`(공유차 실시간: 위치·충전·운행자), `ocrStore`(추출값·확인 대기), `uiStore`
- **서버 상태 vs 클라이언트 상태 분리**: 정비 이력·시세·인프라 검색은 **TanStack Query**, OCR 추출값·낙관적 기록은 Zustand
- **Agent 실행 상태머신**: `idle → planning → tool_running(tool) → streaming → done/error`
- **실시간 이벤트 상태**: 공유차 상태(`car.location`, `car.charging`, `car.in_use`) 이벤트를 받아 `fleetStore`에 reduce
- **다중 사용자 동기화**: 가족 구성원 여러 명이 같은 차의 상태를 보는 동기화, 운행자 충돌(동시 사용 표시) 해소
- **예측 상태 모델링**: 소모품별 임계 주행거리·경과일 규칙 상태(엔진오일 1만km/6개월 등)
- **캐싱 전략**: 시세·인프라 검색 결과 캐시 키 설계, staleTime/gcTime 튜닝

### 4.2 ✅ 목표2 — WebView / 모바일 연동 (이 안의 최대 강점)
- **인앱 웹뷰** 전제(Android/iOS WebView)
- **Native Bridge**: `window.AppBridge` 추상화로 카메라/위치/푸시/햅틱/safe-area 호출
- **카메라 OCR(서비스의 본질)**: 계기판 주행거리·정비명세서·보험증서 촬영 → 비전 OCR 입력
- **위치**: 현재 위치 → 주변 정비소/충전소·공유차 위치
- **Push Notification**: 소모품 교체·리콜·충전 완료·보험 만기 알림
- **Deep Link**: 알림 탭 → 해당 차량/정비 카드로 직접 진입
- **햅틱**: OCR 인식 성공/실패 피드백
- **웹뷰 특수 문제 해결 스토리**: 카메라 권한·해상도, 토큰 주입(Native→Web), 키보드 가림, 뒤로가기 제어, 지도 제스처 충돌 → **핵심 기술 포인트**

> 브릿지 없는 브라우저에서는 Web Camera API/Geolocation/Web Push 폴백으로 동작하게 추상화 → "환경 분기 설계" 경험.

### 4.3 ✅ 목표3 — CS 연결 성능 최적화
- **리스트 가상화**: 차량 기록 피드·정비 타임라인을 `@tanstack/virtual`로 가상화
- **이미지 최적화**: 영수증/명세서/차량 사진 썸네일·원본 분리, lazy load, blur placeholder, 포맷 최적화
- **지도 최적화**: 정비소/충전소 마커 클러스터링·뷰포트 컬링, 경로 렌더 최적화
- **메모이제이션**: 기록 카드·실시간 상태 위젯 `React.memo`, 공유차 이벤트 수신 시 **변경된 위젯만 리렌더** 격리
- **코드 스플리팅 / Lazy Loading**: 지도·OCR 결과·통계 뷰 동적 import
- **Web Worker**: OCR 후처리(텍스트 정규화·항목 파싱) 등 무거운 연산 분리 후보
- **네트워크 최적화**: 검색 debounce/취소(AbortController), 실시간 재연결 백오프, 이미지 업로드 병렬
- **측정→개선 루프**: Lighthouse / React Profiler / web-vitals로 before/after 수치화

### 4.4 ✅ 목표4 — Agent 기반 서비스 (단순 API 호출 X)
- 에이전트가 **도구를 선택·순차/병렬 호출**하는 멀티스텝 구조
- 도구 세트: `ocrDocument`(OCR 추출), `predictMaintenance`(이력·주행거리→교체 시점), `findNearbyService`(주변 정비소/충전소), `getRoute`(경로/이동시간), `checkRecall`(리콜 조회), `estimateResaleValue`(중고 시세)
- 요청 분석 → 계획 수립 → 도구 호출 → 결과 취합 → 최종 응답의 **루프**(예: "오일 갈 때 됐어?" → 이력 조회 → 주기 계산 → 정비소 검색 → 답변)
- (AI 서브 담당) 프론트는 이 과정을 **이벤트 스트림으로 받아 투명하게 반영**

### 4.5 ✅ 목표5 — AI UX / 에이전트 인터페이스
- **Chat UI + Streaming Response**: 토큰 단위 스트리밍
- **에이전트 진행 시각화**: "명세서 OCR → 주행거리 추출 → 교체 주기 계산 → 정비소 3곳 검색" 타임라인/스텝퍼
- **OCR 확인 카드(차별 포인트)**: 추출값을 사용자가 **확인/수정**하는 인터랙션(신뢰·정확도)
- **Tool 실행 과정 표시**: 각 도구 입력·결과를 접이식 카드로 노출
- **실시간 피드백**: 도구 중간 결과 점진 표시, 사용자 중간 개입
- **긴 응답 최적화**: 가상화 + 점진 마크다운 렌더
- **세션/스레드 관리**: 차량별 스레드 분리, 기록 히스토리 복구

## 6. 상태 아키텍처 설계 (핵심 차별화)

```
[서버 상태 — TanStack Query]          [클라이언트 상태 — Zustand]
 - 정비 이력/시세/인프라 검색 (캐시 키)  - garageStore: 차량·소모품 상태·예측
 - 리콜 정보                            - agentRunStore: 실행 상태머신(현재 step/tool)
 - 차량/사용자 메타                     - fleetStore: 공유차 실시간(위치·충전·운행자)
        │                                 - ocrStore: 추출값·확인 대기
        └──────── 동기화 경계 ─────────────┘  - uiStore: 패널·모달·바텀시트
                    ▲
   [실시간 레이어 — WS/SSE 이벤트 → reducer → fleet/garage에 반영]
   car.location / car.charging / car.in_use / agent.delta
```

**에이전트 실행 상태머신**
```
idle ──user msg/사진──▶ planning ──tool 선택──▶ tool_running(tool, args)
   ▲                                              │ 결과 stream
   │                                              ▼
 done ◀── 최종 응답 streaming ◀── aggregating ◀── (도구 더 필요? 반복)
   └─────────────── error/cancel(사용자 중단) ◀──────┘

[OCR 경로] capture → ocrDocument → 추출값 → 사용자 확인/수정 → garageStore 반영
```
> OCR 확인 인터랙션 + 공유차 실시간 동기화가 "복잡한 비동기 상태"의 핵심 설명거리.

## 7. 성능 최적화 — 측정 기반 개선 계획

| 영역 | 측정 지표 | 개선 기법 | 기대 서사 |
|------|----------|----------|----------|
| 첫 진입 | LCP, TTI | SSR + 코드 스플리팅 | "초기 로딩 N초 → M초" |
| 기록 피드 | 스크롤 FPS, DOM 노드 | 가상화 + 이미지 lazy | "대량 기록 부드러운 스크롤" |
| 지도 | 마커 렌더 FPS | 클러스터링 + 뷰포트 컬링 | "다수 인프라 마커 끊김 제거" |
| 공유차 위젯 | 리렌더 횟수 | 메모이제이션 + 이벤트 격리 | "상태 변경당 리렌더 X→1" |
| OCR 후처리 | 메인스레드 블로킹 | Web Worker | "인식 중 입력 지연 해소" |
| 네트워크 | 요청 수/취소 | 검색 캐싱 + AbortController | "중복 호출 제거" |

---

## 8. 팀 역할 분담 (제안)

- **나 (FE 주력 + AI 서브)**: 에이전트 채팅 UI·**OCR 확인 인터랙션**, 상태 아키텍처, 웹뷰 카메라 브릿지, **공유차 실시간 동기화**, 성능 최적화 / AI는 도구 스키마·오케스트레이션 흐름 설계
- FE 보조: 정비 타임라인·지도·기록 피드 UI, 디자인 시스템
- BE: 에이전트 백엔드(LLM 오케스트레이션), 실시간 서버, 정비/시세/리콜 API
- (인원에 따라) 데이터/AI: OCR·비전, 정비 예측 규칙·모델

## 9. 마일스톤 (제안, 약 12주)

1. **W1–2 설계**: 상태 아키텍처·카메라/OCR 브릿지 인터페이스·실시간 이벤트·에이전트 프로토콜 확정
2. **W3–5 코어**: 에이전트 채팅 + 스트리밍 + 도구 시각화 + **OCR 캡처→추출→확인** 플로우
3. **W6–7 정비/예측**: 정비 타임라인·소모품 예측·리콜 조회·예측 푸시
4. **W8 인프라/지도**: 주변 정비소/충전소 검색·경로·마커 최적화
5. **W9 실시간 공유차**: 가족 공유차 상태 동기화 + 다중 사용자 충돌 표시
6. **W10 성능**: 가상화·이미지·코드스플리팅·(OCR 후처리 Worker), 측정→개선 지표 확보
7. **W11 품질**: 테스트·오프라인 재동기화·에러 바운더리
8. **W12 마감**: 배포, 데모, 프로젝트 회고 정리

## 10. 핵심 기술 도전 과제

- **웹뷰 환경 문제 해결** → 카메라 OCR 브릿지, 권한·해상도·토큰 주입·딥링크
- **복잡한 비동기 상태 설계** → 에이전트 상태머신 + 공유차 실시간 동기화 + OCR 확인 + 서버/클라 분리
- **다중 사용자 동기화** → 가족 공유차 운행 충돌 표시·낙관적 업데이트
- **성능 병목 측정·개선** → 기록 가상화·지도 마커·이미지 before/after
- **AI UX 전달** → 에이전트 진행·OCR 확인·도구 근거 시각화
- **모빌리티/차량 도메인 모델링** → 정비 예측·리콜·시세
- **AI 개발 도구(Claude Code/Cursor) 활용** → 개발 과정 기록

## 11. 리스크 & 대응

| 리스크 | 대응 |
|------|------|
| 실차 데이터/OBD 연동 난도 | 코어는 **수동 입력 + OCR**, OBD/커넥티드는 범위 밖(확장 과제) |
| OCR 정확도 | **사용자 확인/수정 카드**로 보정, 햅틱·신뢰도 표시 |
| 정비 예측 모델 난도 | 규칙 기반(주행거리·기간 임계)으로 시작, 모델은 도전 과제 |
| 범위 과다(기능 욕심) | 코어=OCR 기록+정비 에이전트, 공유차 실시간은 2순위 |
| 웹뷰 실기기 테스트 환경 | 브릿지 추상화 + 브라우저 폴백, 실기기 테스트 마일스톤 고정 |
| AI/지도 비용·지연 | 검색 결과 캐싱, 스트리밍으로 체감 지연 완화, 모델 분기 |
