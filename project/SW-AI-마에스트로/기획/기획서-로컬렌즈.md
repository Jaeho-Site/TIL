## 0. 한 줄 요약

> **발품 팔 때 쓰는 똑똑한 AI 현장 수첩.** 사용자가 부동산 임장·중고차 매장·팝업스토어 같은 **오프라인 현장**에 가서, 인앱 웹뷰 앱으로 **위치를 잡거나 사진을 찍으면**, AI 에이전트가 **다중 도구(지도·검색·비전·OCR·RAG)를 스스로 골라 단계적으로 호출**해 실거래가·거주 후기·실시간 반응 등 파편화된 정보를 **하나의 맥락(동적 리포트·체크리스트)** 으로 엮어 준다. 지하처럼 네트워크가 약한 곳에서 쓴 메모·사진은 로컬에 보관됐다가 복구 시 동기화된다.

> **세 안과의 관계**: 로컬렌즈는 스터디파일럿(Web Worker·RAG), 마이카메이트(카메라 OCR·위치), 트립크루(SSR·지도)의 강점을 **"현장 리서치"라는 단일 컨텍스트**로 수렴시킨 통합안이다. "현장의 위치+사진(멀티모달)을 AI가 분석"이라는 한 줄로 5대 목표를 모두 자연스럽게 끌어온다.

## 1. 왜 이 주제인가 — 기술적 의의

`기획요구사항.md`의 5대 목표를 **한 주제로 동시에** 충족한다. 핵심은 "**오프라인 현장의 컨텍스트(위치·사진)를 멀티모달 에이전트가 분석**"이라는 설정이 서로 다른 기술 영역을 한 번에 끌어온다는 점이다.

| 기술 영역 | 핵심 도전 | 대응 요소 |
|------|------|------|
| 웹뷰 / AI | 인앱 웹뷰의 한계를 극복하고 AI 네이티브 경험 설계 | JS-Native 브릿지(카메라·GPS), 로컬(지역) 도메인, AI 도구 활용 |
| 성능 / CS | Next.js SSR + 대용량 처리 + CS 지식 적용 | **Web Worker 파싱 스레드 분리**, 가상화, SSR/ISR(공유 리포트 SEO), Lighthouse |
| 상태 / 아키텍처 | 복잡한 비동기 상태를 구조적으로 관리·확장 | Zustand/TanStack **에이전트 상태 전이 파이프라인**, Zod 스키마 검증, FSD 아키텍처 |

> **결론**: 단일 프로젝트로 **세 가지 다른 기술 관점**(웹뷰/AI, 성능/CS, 상태/아키텍처)을 모두 깊게 다룰 수 있다. "세상에 없던 혁신"이 아니라 **"기존 불편을 LLM·RAG·비전 조합으로 얼마나 매끄러운 FE UX로 풀었는가"** 를 보여주는 데 최적화돼 있다.

## 2. 서비스 개요

### 2.1 문제 정의
오프라인 현장(임장·중고차·답사)에서는 **즉각적 정보 탐색과 기록**이 필요한데, 사용자는 (1) 호갱노노·직방·헤이딜러·스마트렌즈 등 **여러 앱을 오가며 수동 검색**해야 하고, (2) 검색·사진·메모가 **흩어져 하나의 판단 자료로 엮이지 않으며**, (3) 지하·실내 등 **네트워크 불안정 환경**에서 기록이 유실된다.

### 2.2 솔루션
앱 안의 **AI 에이전트**가 사용자의 의도(Intent)를 파악해 **지도·검색·비전·OCR·RAG 도구를 순차/병렬 호출**하여 결과물을 조립한다. 기존 앱들이 "수동 DB 조회"라면, 로컬렌즈는 **현장 컨텍스트 기반 능동 리서치 + 나만의 동적 리포트 생성**이다.

- **현장 분석 플로우(에이전트 주도)**: 위치/사진 입력 → 주소·건축물대장 조회 → 실거래가·시세 계산 → 커뮤니티/블로그 후기 검색 → 교통 도보시간 계산 → **요약 리포트 + 체크리스트 생성**
- **비전 분석 플로우(멀티모달)**: 방 내부/문서 촬영 → 이미지 분석(곰팡이·채광·결로 의심) 또는 OCR(가격표·성능점검기록부) → 체크리스트 자동 추가
- **오프라인 기록 플로우**: 네트워크 약한 곳에서 작성한 메모·사진을 IndexedDB에 보관 → 복구 시 서버 동기화

### 2.3 사용자 가치
- 현장에서 **버튼 한 번**으로 흩어진 정보를 한 맥락으로
- 사진(비전)으로 **눈에 보이는 상태**까지 분석·기록
- 약한 네트워크에서도 **기록이 유실되지 않는다**

## 4. 기획요구사항(`기획요구사항.md`) 5대 목표 매핑

### 4.1 ✅ 목표1 — 복잡한 상태관리 (Zustand + TanStack Query)
- **서버 상태(TanStack Query)**: 외부 API·자체 백엔드(예: Supabase)의 장소·실거래가·리뷰 데이터 캐싱, 무효화, 백그라운드 업데이트
- **클라이언트·에이전트 상태(Zustand)**: 에이전트 다단계 상태머신 `idle → tool 선택 → 외부 API 호출 중 → 데이터 분석 중 → 응답 생성 중` 을 전역 관리, UI에 1:1 매핑
- **오프라인/동기화 상태**: 현장(지하 등) 작성 메모·체크리스트·사진을 **IndexedDB**에 저장 후 네트워크 복구 시 서버와 동기화(충돌은 last-write-wins + 변경 큐 리베이스)
- **캐싱 전략**: 동일 좌표·동일 건물 분석 결과 캐시 키 설계, staleTime/gcTime 튜닝, 도구 재호출 시 캐시 재사용
- **스토어 분리(제안)**: `agentRunStore`(실행 상태머신), `reportStore`(리포트·체크리스트), `captureStore`(사진·OCR 추출값), `syncStore`(오프라인 큐), `uiStore`

### 4.2 ✅ 목표2 — WebView / 모바일 연동 (JS-Native Bridge)
- **인앱 웹뷰** 전제 (React Native/Flutter 앱 셸 + 웹뷰)
- **Native Bridge 인터페이스 설계**: `window.AppBridge` 추상화 레이어
  - `requestCamera()`: 현장 사진 촬영 → 이미지 전송(비전/OCR 입력)
  - `getLocation()`: 현재 GPS 좌표 기반 주변 데이터 탐색
  - `sendPushNotification()`: 긴 분석 작업 완료 시 웹뷰 밖 사용자에게 푸시
  - `(추가)` 햅틱·safe-area·토큰 주입(Native→Web)
- **Deep Link**: 푸시 탭 → 특정 현장 리포트로 직접 진입
- **웹뷰 특수 문제 해결 스토리**: 카메라 권한·해상도, 메모리 누수/초기 렌더링 병목, 키보드 가림, 뒤로가기 제어 → **핵심 기술 포인트**

> 브릿지 없는 일반 브라우저에서는 Web Camera API/Geolocation/Web Push 폴백으로 동작하게 추상화 → "환경 분기 설계" 경험.

### 4.3 ✅ 목표3 — CS 연결 성능 최적화
- **Web Worker**: 에이전트가 반환하는 **대용량 JSON·긴 마크다운 파싱**을 메인 스레드에서 분리 → 렌더링 블로킹(TBT/Long Task) 방지
- **렌더링 최적화**: 실시간으로 길어지는 채팅·리포트에 **Virtualization** 적용 → DOM 노드 최소화. 메시지/리포트 블록 `React.memo` + 안정적 key, 스트리밍 중 리렌더 격리
- **Next.js SSR/ISR**: 공유용 완성 리포트는 SEO·OG를 위해 **SSR/ISR**로 서빙해 초기 LCP 최적화
- **코드 스플리팅 / Lazy Loading**: 지도·비전 결과·PDF/문서 뷰어 동적 import
- **이미지 최적화**: 현장 사진 썸네일/원본 분리, lazy load, blur placeholder, 포맷 최적화
- **네트워크 최적화**: 스트리밍(SSE) 청크, 검색 debounce/취소(AbortController), 재시도 백오프
- **측정→개선 루프**: Lighthouse / React Profiler / web-vitals로 **TBT·LCP·FPS before/after** 수치화

### 4.4 ✅ 목표4 — Agent 기반 서비스 (ReAct 패턴)
- **Vercel AI SDK / LangChain**로 **ReAct(Reasoning + Acting)** 패턴 구현 — 단일 프롬프트 응답이 아니라 LLM이 목적에 따라 Tool을 스스로 선택
- **Tool Calling 시나리오(부동산 예)**:
  1. 사용자 사진/위치 입력
  2. `getAddress`(Map API)로 좌표→주소·건축물대장
  3. `searchMarketPrice`(실거래가/시세)
  4. `searchCommunity`(블로그·동네생활 거주 후기)
  5. `getTransit`(가까운 역 도보시간)
  6. `analyzeImage`/`ocrDocument`(방 상태/문서)
  7. `retrievePastReports`(Vector DB RAG로 과거 유사 임장 기록)
  8. 결과 취합 → 리포트 구조화
- 사용자 의도 분석 → 계획 → 도구 호출 → (부족 시 재호출 반복) → 취합 → 최종 리포트의 **루프**

### 4.5 ✅ 목표5 — AI UX / 에이전트 인터페이스
- **Thought Process UI**: 로딩 스피너 대신 현재 실행 중인 Tool·상태를 **아코디언으로 실시간 시각화**
  - 📍 "현재 위치 기반 건축물대장 조회 중…"
  - 💰 "최근 3개월 실거래가·평균 월세 계산 중…"
  - 🗣️ "동네 커뮤니티·블로그 거주 후기 30건 요약 중…"
  - 🚇 "가장 가까운 지하철역 도보 시간 계산 중…"
- **Streaming Response**: 생성 리포트를 청크 단위 스트리밍 렌더 → 체감 대기 단축
- **Tool 실행 과정 표시**: 각 도구 입력·결과를 접이식 카드로 노출(근거·투명성)
- **OCR/비전 확인 카드**: 추출값·분석 결과를 사용자가 확인/수정
- **세션/스레드 관리**: 현장별 리포트 스레드 분리, 히스토리 복구

## 6. 상태 아키텍처 설계 (핵심 차별화)

```
[서버 상태 — TanStack Query]          [클라이언트 상태 — Zustand]
 - 장소/실거래가/리뷰/시세 (캐시 키)     - agentRunStore: 실행 상태머신(현재 step/tool)
 - RAG 과거 리포트 검색                 - reportStore: 리포트·체크리스트·사진 첨부
 - 건축물대장/교통 등 도구 결과          - captureStore: 사진·OCR/비전 추출값
        │                                 - syncStore: 오프라인 큐(IndexedDB 미러)
        └──────── 동기화 경계 ─────────────┘  - uiStore: 패널·아코디언·바텀시트
                    ▲                                   ▲
   [스트리밍 레이어 — SSE: agent.delta/tool.start/tool.result → reduce]
   [오프라인 레이어 — IndexedDB ⇄ syncStore, 복구 시 서버 머지(LWW)]
   [Web Worker — 대용량 JSON/마크다운 파싱 → 메인스레드 비차단]
```

**에이전트 실행 상태머신 (ReAct)**
```
idle ──위치/사진/요청──▶ reasoning(계획) ──tool 선택──▶ tool_running(tool, args)
   ▲                                                      │ 결과/관찰(observation)
   │                                                      ▼
 done ◀── 리포트 streaming ◀── aggregating ◀── (목표 달성? 아니면 재계획 반복)
   └──────────────── error/cancel(사용자 중단) ◀────────────┘
```
> Thought Process 아코디언 UI를 이 상태머신과 **1:1 매핑**하는 것이 "AI UX" 핵심 구현 포인트.

| 순위 | 도메인 | 핵심 Tool 조합 | 선정 이유 |

| **1 (코어)** | **부동산 임장** | 주소·실거래가·거주후기·교통·방상태 비전 | 시나리오가 가장 구체적, 도구 다양성↑, RAG(과거 임장) 자연스러움 |

## 부록 B. 동네비서·자매 안과의 관계
- **수렴**: 로컬렌즈는 스터디파일럿(Worker·RAG)·마이카메이트(OCR·위치)·트립크루(SSR·지도)의 강점을 "현장 리서치"로 통합한 안.
- **차별(vs 동네비서)**: 동네비서가 "거래(등록·탐색·1:1 채팅)"라면, 로컬렌즈는 **"현장 의사결정 리서치(분석·리포트)"** — 멀티모달 입력과 ReAct 도구 오케스트레이션, 오프라인 동기화가 중심.
- **선택 기준**: 단일 프로젝트로 세 기술 영역(웹뷰/AI·성능/CS·상태/아키텍처)을 **균형 있게** 가져가고, 선호 스택(Next 15·TanStack·Zod·FSD)을 전면에 쓰고 싶다면 로컬렌즈가 통합 최적안.
