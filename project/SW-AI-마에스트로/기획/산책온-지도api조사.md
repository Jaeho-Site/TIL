한줄평 : 우리 서비스는 보행을 다루는 서비스다보니 T MAP이 가장 어울리는 api인듯합니다. (우연히도 SK?) 이유는 아래표에 나와있고, AI가 만든 표이긴하지만 사실여부는 어느정도 확인한 상태입니다. 하지만 써봐야 안다고, TMAP으로 쓰다가 안좋으면 바꾸고 이런식으로 가는게 맞는듯합니다. 

API의 한계는 딱히 보이지 않는게, 일단은 공공데이터를 사용하기로 했고, 우리 서비스에서 지도의 역할은 시간계산이나 장소검색, 공공데이터의 장소들을 경로표시해주는 역할 정도여서 딱히 한계라고 할것은 없어보입니다. (서비스 자체가 공공데이터와 AI에 대한 의존도가 높은것이고, 지도 자체에 대한 의존성은 낮기 때문)

!3대 지도조사.png

- Naver 지도 api는 Maps라는 지도만 지원하는 api로 바뀐다고함. 추가된 기능은 기술적인것은 거의 없는것 같고 지도 꾸미기 등 개발자 커스텀 기능 정도 추가된듯.
- 일단 3대 지도 api중에 보행 경로를 다루는건 T Map이 유일함.

#### 1. 장소 검색

사용자가 "30분 산책", "조용한 분위기", "공원 위주"와 같은 조건을 입력하면 AI는 우선 주변의 산책 가능 장소를 탐색해야 한다. 근데 우린 공공 API사용해서 AI가 처음부터 경로를 생성해줄 필요는 없으니 안쓸수도? 근데 일단 장소 검색은 잘되는듯.

TMAP 장소 검색 API를 활용하면 다음과 같은 주요 지점을 수집할 수 있다.

- 공원, 산책로,하천변, 광장, 카페거리, 관광 명소

예를 들어 대전역 주변에서 산책 코스를 생성한다면 다음과 같은 후보를 확보할 수 있다.

> 대전역 → 갑천 산책로 → 한밭수목원 → 엑스포시민광장
> 

AI는 이러한 장소들 중 사용자 조건에 가장 적합한 후보를 선별하고, 이후 경로 생성 단계로 전달한다.

#### 2. 경로 생성 및 거리 계산

산책온의 핵심 기능은 단순 목적지 안내가 아니라 "산책 코스 생성"이다. AI가 추천한 장소들을 실제 보행 가능한 동선으로 연결하기 위해 TMAP 보행자 경로 API를 활용한다. 보행 경로를 지원한다는게 좋은듯.

```
출발지
↓
갑천 산책로
↓
한밭수목원
↓
출발지
```

TMAP은 각 구간을 실제 보행 네트워크 기준으로 계산하여 다음 정보를 제공한다.

- 총 이동 거리
- 세부 경로 좌표
- 횡단보도 및 보행로 정보
- 예상 소요시간

이를 통해 개발팀이 별도의 경로 탐색 알고리즘(Dijkstra, A*)을 직접 구현하지 않아도 실제 이동 가능한 산책 코스를 생성할 수 있다.

#### 3. 예상 소요시간 계산

산책온은 사용자가 입력한 "시간"을 기준으로 코스를 추천해야 한다. TMAP 보행자 경로 API는 경로 계산 결과와 함께 예상 소요시간을 제공하므로 이를 추천 필터로 활용할 수 있다.

| 사용자 요청 | 추천 기준 |
| --- | --- |
| 20분 산책 | 15~25분 코스 |
| 30분 산책 | 25~40분 코스 |
| 1시간 산책 | 50~70분 코스 |

#### 4. 산책 진행 및 완주 관리

산책 중에는 GPS 위치와 추천 경로를 비교하여 사용자의 진행 상황을 확인할 수 있다.

- 현재 위치 표시
- 남은 거리 표시
- 남은 시간 표시
- 경로 이탈 감지
- 재탐색 기능 제공
- 주요 지점 방문 여부 확인

특히 기획서에 명시된 "주요 지점 기반 산책 진행" 기능과 직접 연결되어 완주 여부를 판단하는 기준으로 활용할 수 있다.

#### TMAP API의 한계와 대안

</aside>

> 한줄평 : 지도 API를 써서 경로/거리/시간/장소 검색을 하는건 모두 가능해보임. 직접적으로 보행을 지원하는 TMAP사용하면 어느정도 다 커버할듯. 정확한 한계는 써봐야 알겠지만, 한계로 지목하는게 다 우리팀이 정의한 한계인 경로품질인데 사실 이건 지도 api의 역할이 아님.
> 

#### 1. 보행 환경 품질 정보 부족

TMAP은 길의 존재 여부는 알려주지만 실제 보행 품질은 알기 어렵다.

- 인도가 매우 좁음
- 야간 조도가 부족함
- 공사 중인 구간
- 실제 체감 안전도가 낮음

**대안**

기획서의 사용자 메모 및 피드백 기능을 적극 활용한다. 사용자가 남긴 의견을

- 어두움, 위험 구간, 공사 중, 보행 불편, 풍경 우수

등의 태그로 변환하고, 이후 추천 점수 계산에 반영한다.

#### 1-1 초기 데이터 부족 문제

서비스 초기에는 사용자 피드백이 없기 때문에 추천 품질이 제한될 수 있다. 어차피 공공데이터를 활용할 예정이니 초기 단계에서는

- 공원, 하천, 산책로, 관광 명소

중심의 추천을 제공하고,이후

- 산책 기록, 사용자 메모, 완주 데이터, 만족도 평가

를 축적하여 추천 품질을 개선하는 방식이 현실적이다.
