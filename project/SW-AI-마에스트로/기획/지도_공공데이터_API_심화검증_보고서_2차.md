# 1. TMAP 보행자 API 심화 조사 (실호출)

**공통 Request (실제 사용)**
```bash
curl -X POST "https://apis.openapi.sk.com/tmap/routes/pedestrian?version=1&format=json" \
 -H "appKey: QrqfJJm26I5kWsKHlI4mA7IX77yOQJ8FasRSmVCL" -H "Content-Type: application/json" \
 -d '{"startX":"<lon>","startY":"<lat>","endX":"<lon>","endY":"<lat>",
      "startName":"S","endName":"E","reqCoordType":"WGS84GEO","resCoordType":"WGS84GEO","searchOption":"0"}'
```
> 총 **50개 이상 경로**를 서울/부산/대전에서 호출해 분석함.

## 1-1. facilityType 실제 반환값 검증

**실제로 반환된 facilityType [실호출]**: `1, 11, 12, 14, 15, 17`
- 분포(대표 표본 11개 경로): `{11: 126회, 15: 36회, 17: 8회, 14: 4회, 1: 2회}` + 타겟 경로에서 `12` 확인
- **`16`(계단)은 끝까지 미관측** (뒤 1-2 참조)

### 도시별 실제 확인 사례 (호출좌표·Response·실제 장소)

**■ 대전 — 육교 (turnType 125 / facilityType 12)** [실호출]
- 호출좌표: `start(127.41000, 36.33000) → end(127.41050, 36.33200)` (계룡로 일대)
- Response(일부):
```json
{ "geometry":{"type":"Point","coordinates":[127.40964969973295, 36.330477185972676]},
  "properties":{ "turnType":125, "facilityType":"12", "pointType":"GP",
                 "description":"육교 진입 후 29m 이동" } }
```
- 실제 장소: 대전 계룡로 횡단 **육교**. → 물리시설 '육교'는 `turnType=125` + description "육교 진입"으로 확정.

**■ 서울 — 지하보도 (turnType 126 / facilityType 14)** [실호출]
- 호출좌표: `start(126.97220, 37.55520) → end(126.96820, 37.55430)` (서울역↔서부역)
- Response(일부):
```json
{ "geometry":{"type":"Point","coordinates":[126.97188585900872, 37.554305985580854]},
  "properties":{ "turnType":126, "facilityType":"14", "pointType":"GP",
                 "name":"서울역 1번출구", "description":"서울역 1번출구에서 지하보도 진입 후 75m 이동" } }
```
- 실제 장소: **서울역 1번출구 지하보도**. → '지하보도'는 `turnType=126` + facilityType 14.

**■ 서울/부산 — 횡단보도 (turnType 211·212·213 / facilityType 15)** [실호출]
- 서울역: `[126.97222, 37.55478]` turnType 213 "우측 횡단보도 후...", `[126.96918, 37.55392]` turnType 212 "좌측 횡단보도 후..."
- 부산 감천문화마을(`129.01040,35.09700→129.01100,35.09820`): turnType 212 횡단보도 관측
- → '횡단보도'는 `turnType=211/212/213`. 이 지점들의 facilityType은 `15`.

**■ facilityType 11 / 17 (일반 보행자도로 구간)** [실호출]
- 남산·낙산 등 대부분 구간에서 facilityType 11, 17이 나오며 LineString의 `name="보행자도로"`, `roadType=21~24`.
- facilityType 17 구간의 표기도 "보행자도로"(=일반 보행자도로)이며 **'경사로'로 표기되지 않음.**

## 1-2. facilityType 문서 vs 실제 응답 비교 (핵심 발견)

ToDo/PDF의 코드표(`11=건물연결통행로, 12=보행자도로, 15=육교, 16=계단, 17=경사로`)는 **실제 API와 상당 부분 불일치**한다. 실호출로 관측된 실제 대응은 다음과 같다.

| 코드 | 문서(ToDo/PDF) 주장 | **실제 관측 [실호출]** | 일치? |
| :---: | --- | --- | :---: |
| 11 | 건물연결통행로 | 가장 흔한 일반값(보행자도로/도로명/건물출구 전반) | ✗ |
| 12 | 보행자도로 | **육교 진입점**(turnType 125)에서 관측 | ✗ |
| 14 | 지하보도 | **지하보도 진입점**(turnType 126)에서 관측 | **✓** |
| 15 | 육교 | **횡단보도**(turnType 211~213)에서 관측 | ✗ |
| 16 | 계단 | **미관측** (아래) | — |
| 17 | 경사로 | 일반 보행자도로 구간(name="보행자도로") | ✗ |

**결론**: facilityType 코드만으로 계단/육교/경사로를 판별하면 **오분류**한다. 물리시설의 **정답 신호는 `turnType`**이다(아래는 실호출로 확인/공식표와 일치).

| turnType | 의미 | 검증 |
| :---: | --- | --- |
| 125 | 육교 | **[실호출]** 대전 계룡로 |
| 126 | 지하보도 | **[실호출]** 서울역 |
| 127 | 계단 진입 | [공식문서] (실호출 미관측) |
| 128 | 경사로 진입 | [공식문서] |
| 218 | 엘리베이터 | [공식문서] |
| 211/212/213 | 횡단보도/좌·우측 횡단보도 | **[실호출]** 서울·부산·대전 다수 |
| 200/201/11/12/13/14 | 출발/도착/직진/좌/우/유턴 | **[실호출]** + [공식문서] |

**보행자도로가 실제로 11인지?** → 아니오. 보행자 전용구간은 LineString의 **`roadType`(21/22/23/24)** 과 `name="보행자도로"`로 표기되며, facilityType은 11이 대부분이다.
**12는 실제로 언제 나오는가?** → 본 조사에서는 **육교 진입 지점**에서 관측되었다(보행자도로 아님).

### ⚠️ 계단(16/turnType 127) 미검출 — MVP 중요 발견 [실호출]
부산 168계단·초량 이바구길·감천 골목계단, 서울 남산·낙산·인왕산 성곽계단, 해방촌 계단 등 **계단이 확실한 8개 지점을 직접 경로로 호출**했으나, `searchOption`을 0/4/10/30으로 바꿔도 **facilityType 16 또는 turnType 127(계단)이 한 번도 반환되지 않음.** TMAP 기본 보행 라우팅이 해당 계단을 우회하거나 라우팅 네트워크에 계단 엣지를 노출하지 않음.
- **함의(관절 보호 시나리오)**: TMAP은 계단을 알아서 회피하는 경향이 있어 "계단 없는 경로"에는 유리하나, **"이 경로에 계단이 있는지"를 TMAP만으로 신뢰성 있게 태깅하기 어렵다.** 계단 회피 보장/표기를 위해서는 보조 데이터(OSM `highway=steps`, AI Hub 노면, 도로대장 경사)가 필요.

## 1-3. MVP에서 사용할 추가 필드 (실응답 기반 추천)

| 필드 | 위치 | MVP 활용 | 검증 |
| --- | --- | --- | :---: |
| **turnType** | Point | ★ 관절보호/안전 핵심 — 육교·지하보도·계단·경사로·횡단보도 감지 | [실호출] |
| **description** | Point | 한글 안내문("육교 진입","지하보도 진입","횡단보도 후") — 태깅·설명 생성용 | [실호출] |
| **roadType / categoryRoadType** | LineString | 보행자도로 등급(21/22/23/24) — 보행 쾌적도 | [실호출] |
| **name** | Point/Line | 도로·시설·건물명("서울역 1번출구","퇴계로") — 코스 설명/POI | [실호출] |
| **totalDistance / totalTime** | 첫 Point | 코스 거리·소요시간(추천 조건 매칭) | [실호출] |
| **pointType** (SP/GP/EP) | Point | 출발/경유/도착 — 주요 지점 동선 구성 | [실호출] |
| **geometry.coordinates** | 전체 | 실제 경로 폴리라인/노드 좌표 | [실호출] |

## TMAP 종합
- **조사 결과**: 키 활성화로 실호출 성공. facilityType 실측값 = 1/11/12/14/15/17.
- **발견한 facilityType**: 11(일반), 12(육교 지점), 14(지하보도), 15(횡단보도 지점), 17(일반 보행자도로). **16(계단) 미검출.**
- **문서와 다른 점**: ToDo/PDF의 facilityType 코드표는 14(지하보도)만 일치, 나머지는 불일치. **물리시설은 turnType이 정답.**
- **MVP 활용 가능성**: **높음.** 경로 좌표·거리·시간 + turnType 기반 시설 태깅으로 안전/관절보호 가중치 설계 가능. 단 **계단 판별은 보조 데이터 필요.**

# 3. MVP 후보 지역 추천 → **대전**

확보 데이터(TMAP·국가숲길·보행자전용도로·국가소음·AI Hub)를 4대 목적(자연친화/조용/관절보호/안전)에 대입한 결과:

**대전을 MVP 시작 지역으로 추천한다.** 근거:
1. **자연친화(국가숲길)**: 3개 후보 도시 중 **대전만 국가숲길(대전둘레산길)을 보유** → 핵심 차별 목적인 '자연친화 산책' 데이터를 유일하게 확보 [공식문서].
2. **안전**: 1차 PDF가 인용한 대전 특화 데이터가 풍부 — **대전 횡단보도조회서비스, 보행자 교통사고 통계(전체 사고의 52%가 보행자, 55%가 교차로 횡단 중)**, 야간 무장애 소음지도. 보행 위험구간 태깅에 직접 활용.
3. **관절보호/보행시설**: TMAP **육교(turnType 125) 실증이 대전에서 확인**됨. 도로대장 종단경사 + DEM으로 경사 보강 가능.
4. **운영 현실성**: 도심 규모가 서울보다 작아 초기 데이터 구축·제한지역 현장테스트 관리가 용이. 기획서상 테스트 지역도 대전/부산. (팀 정체성 '성심당' = 대전)

**보조 순위**: 서울(보행 인프라·소음지도 최다 → 확장 2순위), 부산(급경사·계단 시연에 좋으나 국가숲길 0 + TMAP 계단 미표기로 관절보호 데이터 공백).

# 최종 결론

1. **TMAP 실제 확인 facilityType**: `1, 11, 12, 14, 15, 17` (16=계단 미검출). 물리시설의 정답 신호는 **turnType** — 실증: **육교(125, 대전)·지하보도(126, 서울역)·횡단보도(211~213, 3도시)**. ToDo/PDF 코드표는 14(지하보도)만 일치.
2. **국가숲길 데이터가 가장 풍부한 지역**: **대전(대전둘레산길)**. 서울·부산 0곳. (API는 현재 502로 POI 상세 미취득, 지정목록·GPX로 검증)
3. **MVP 시작 최적 지역**: **대전.**
4. **추가로 확보하면 좋은 데이터**
   - **계단/경사(관절보호)**: OSM `highway=steps`·`incline`, AI Hub 인도보행 노면/Depth, 도로대장 종단경사 + DEM (TMAP 계단 공백 보완)
   - **도심 자연산책로**: 두루누비 걷기여행길 API, 산림청 숲나들e, 서울둘레길/부산갈맷길 (국가숲길 도심 공백 보완)
   - **소음 좌표화**: 국가소음 도로지점정보(주소→지오코딩) 또는 도로교통소음_격자(59394979)로 좌표 확보
   - **국가숲길 API**: 제공기관(1566-0025) 502 장애 복구 확인 후 POI 재수집
