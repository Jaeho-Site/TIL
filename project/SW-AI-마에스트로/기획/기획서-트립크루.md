## 0. 한 줄 요약

> **친구들과 같은 화면에서 함께 짜는 여행 일정표.** 사용자가 "3박4일 후쿠오카, 미식 위주, 1인 60만원"이라고 말하면, 에이전트가 **여러 도구(항공·맛집·동선·날씨·예산)를 단계적으로 호출**해 일정 초안을 만들고, **여러 명이 실시간으로 동시에 카드를 드래그·수정**하면 그 변경이 모두에게 즉시 동기화된다. 동네비서가 "1:1 거래 채팅"이라면, 트립크루는 **N명 실시간 협업 편집**이라는 더 어려운 멀티유저 문제를 정면으로 다룬다.

## 1. 왜 이 주제인가 — 기술적 의의

`기획요구사항.md`의 5대 목표를 **한 주제로 동시에** 충족하되, 동네비서가 다루지 않은 **검색·지도·SSR·대용량·실시간 협업** 영역을 정조준한다.

| 핵심 기술 영역 | 본 프로젝트의 대응 요소 |
|------|------|
| React+TS, SSR/성능 최적화, 대용량 리스트, 검색·지도 | 장소/항공 **검색**, 지도 대량 마커, 일정 공유 링크 **SSR**, 리스트 가상화 |
| 대용량 트래픽·BFF | 검색 결과 캐싱·BFF형 도구 게이트웨이 |
| 인앱 웹뷰 + Native Bridge + AI 도구 | 인앱 웹뷰 전제 + 위치·카메라·푸시·딥링크 브릿지 |
| 실시간·다중 사용자·동기화 | **N명 실시간 협업 편집 + presence + 충돌 해소** |
| 상태관리, 브라우저 동작 원리 | Zustand 복합 상태 + 렌더링 최적화 |

> **결론**: 동네비서가 웹뷰+1:1 거래 채팅을 강점으로 한다면, 트립크루는 **검색·지도·SSR·대용량**을 정면으로 다루면서 공통 기반(React+TS+상태관리+웹뷰)까지 포섭한다. 특히 **실시간 동시 편집의 충돌 해소**는 1:1 채팅보다 난도가 높아 "복잡한 비동기 상태" 서사를 가장 강하게 만든다.

## 2. 서비스 개요

### 2.1 문제 정의
여행 계획 시 사용자는 **(1) 정보 수집(항공·숙소·맛집·동선·날씨·예산)이 막막하고, (2) 친구들과 카톡으로 합의하면 일정이 흩어지고 충돌**한다. "누가 어떤 버전을 갖고 있는지" 모호하고, 동선·예산이 깨진다.

### 2.2 솔루션
앱 안의 **AI 에이전트**가 자연어 요청을 받아 도구를 골라 **일정 초안**을 생성하고, **공유 일정 보드**에서 여러 사용자가 실시간 공동 편집한다. 에이전트는 편집 중에도 "이 동선이면 이동 2시간, 대안 추천"처럼 실시간 보조한다.

- **생성 플로우(에이전트 주도)**: "3박4일 후쿠오카, 미식, 1인 60만원" → 항공 검색 → 맛집/명소 검색 → 동선·이동시간 계산 → 예산 추정 → 일자별 일정 초안 작성 → 보드에 카드로 배치
- **협업 플로우(다중 사용자)**: 친구들이 같은 보드에서 카드 드래그/추가/삭제, 서로의 커서·선택이 실시간 표시, 충돌은 규칙으로 해소
- **현장 플로우(웹뷰)**: 여행 중 위치 기반 "근처 추천", 영수증/티켓 카메라 기록, 일정 시작 푸시 알림

### 2.3 사용자 가치
- 초보자도 **말 한마디로** 일정 초안을 얻는다
- 친구들과 **흩어지지 않고 한 화면에서** 합의한다
- 에이전트가 **무슨 근거로 이 동선을 짰는지 투명하게** 보여줘 신뢰 형성

## 4. 기획요구사항(`기획요구사항.md`) 5대 목표 매핑

### 4.1 ✅ 목표1 — 복잡한 상태관리 (이 안의 최대 강점)
- **Zustand 전역 스토어 분리**: `planStore`(일정 보드: 일자/카드/순서), `agentRunStore`(에이전트 실행 상태머신), `presenceStore`(참여자 커서·선택·읽음), `uiStore`(패널·바텀시트)
- **서버 상태 vs 클라이언트 상태 분리**: 검색 결과(항공·장소·날씨)는 **TanStack Query**, 보드 편집 상태·낙관적 변경은 Zustand
- **다중 사용자 동기화(핵심 난이도)**: 동시 편집 충돌 해소를 **단계적**으로 설계
  - 1단계: **카드 단위 잠금 + 낙관적 업데이트** + 서버 권위 머지(버전 번호/last-write-wins)
  - 2단계(도전): 순서 변경 충돌에 **간이 OT/CRDT**(fractional indexing으로 카드 순서 표현) 적용
- **실시간 이벤트 상태**: WebSocket 이벤트(`card.moved`, `card.added`, `presence.update`)를 받아 reducer로 보드 상태 reduce, 자기 변경은 echo 무시
- **Agent 실행 상태머신**: `idle → planning → tool_running(tool) → streaming → done/error`
- **캐싱 전략**: 동일 도시·기간 검색 결과 캐시 키 설계, staleTime/gcTime 튜닝, 도구 재호출 시 캐시 재사용

### 4.2 ✅ 목표2 — WebView / 모바일 연동
- **인앱 웹뷰** 전제(Android/iOS WebView)
- **Native Bridge**: `window.AppBridge` 추상화 레이어로 위치/카메라/푸시/햅틱/safe-area 호출
- **위치**: 현재 위치 → 주변 명소/맛집 추천, 여행 중 동선 보정
- **카메라**: 영수증·티켓·현장 사진을 일정 카드에 첨부
- **Push Notification**: 일정 시작/예약 시간/항공 체크인 알림
- **Deep Link**: 푸시 탭 → 특정 일정 카드·보드로 직접 진입
- **웹뷰 특수 문제 해결 스토리**: 키보드 가림, iOS 바운스 스크롤, 토큰 주입(Native→Web), 뒤로가기 제어, 지도 제스처와 웹뷰 스크롤 충돌 → **핵심 기술 포인트**

> 브릿지 없는 일반 브라우저에서는 Web Geolocation / Web Push / 파일 input 폴백으로 동작하게 추상화 → "환경 분기 설계" 경험.

### 4.3 ✅ 목표3 — CS 연결 성능 최적화
- **지도 대량 마커**: 수백~수천 마커 **클러스터링** + 뷰포트 컬링, 마커 재사용
- **리스트 가상화**: 장소 탐색 피드를 `@tanstack/virtual`로 가상화
- **메모이제이션**: 보드 카드 컴포넌트 `React.memo` + 안정적 key, 협업 이벤트 수신 시 **변경된 카드만 리렌더** 격리
- **협업 이벤트 throttle/배치**: presence(커서) 이벤트를 throttle, 보드 변경은 배치 적용
- **코드 스플리팅 / Lazy Loading**: 지도·통계·보드 뷰 동적 import
- **이미지 최적화**: 장소 썸네일/원본 분리, lazy load, blur placeholder
- **네트워크 최적화**: 스트리밍(SSE) 청크, 검색 요청 debounce/취소(AbortController), 재연결 백오프
- **측정→개선 루프**: Lighthouse / React Profiler / `web-vitals`로 before/after 수치화

### 4.4 ✅ 목표4 — Agent 기반 서비스 (단순 API 호출 X)
- 에이전트가 **도구를 선택·순차/병렬 호출**하는 멀티스텝 구조
- 도구 세트: `searchFlights`(항공), `searchPlaces`(명소/맛집), `getRoute`(동선/이동시간), `getWeather`(날씨), `estimateBudget`(예산), `buildItinerary`(일자별 배치)
- 요청 분석 → 계획 수립 → 도구 호출 → 결과 취합(동선 최적화) → 일정 초안 생성의 **루프**
- (AI 서브 담당) 프론트는 이 과정을 **이벤트 스트림으로 받아 UI에 투명하게 반영**

### 4.5 ✅ 목표5 — AI UX / 에이전트 인터페이스
- **Chat UI + Streaming Response**: 토큰 단위 스트리밍
- **에이전트 진행 시각화**: "항공 검색 중 → 맛집 12곳 조회 → 동선 최적화 중 → 일정 배치 중" 타임라인/스텝퍼
- **Tool 실행 과정 표시**: 각 도구 입력·결과를 접이식 카드로 노출(왜 이 동선인지 근거 제공)
- **실시간 피드백**: 도구 중간 결과 점진 표시, 사용자가 중간 개입(취향 추가·재계획)
- **긴 응답 최적화**: 가상화 + 점진 마크다운 렌더
- **세션/스레드 관리**: 여행별 보드 분리, 생성 히스토리 복구
- **Workflow 시각화**: 어떤 도구가 왜 호출됐는지 보이는 뷰

## 6. 상태 아키텍처 설계 (핵심 차별화)

```
[서버 상태 — TanStack Query]          [클라이언트 상태 — Zustand]
 - 항공/장소/날씨 검색 결과 (캐시 키)    - planStore: 일자·카드·순서(fractional index)
 - 일정 보드 스냅샷/히스토리            - agentRunStore: 실행 상태머신(현재 step/tool)
 - 사용자/여행 메타                     - presenceStore: 참여자 커서·선택·읽음
        │                                  - uiStore: 패널·모달·바텀시트
        └──────── 동기화 경계 ──────────────┘
                    ▲
   [실시간 레이어 — WS 이벤트 → reducer → 보드/presence에 반영]
   card.added / card.moved / card.removed / presence.update / agent.delta
```

**에이전트 실행 상태머신**
```
idle ──user msg──▶ planning ──tool 선택──▶ tool_running(tool, args)
   ▲                                          │ 결과 stream
   │                                          ▼
 done ◀── 일정 배치 streaming ◀── aggregating ◀── (도구 더 필요? 반복)
   └─────────────── error/cancel(사용자 중단) ◀──────┘
```

**협업 충돌 해소 규칙(요지)**
```
- 카드 이동: fractional index 재계산 → 동시 이동 시 서버 버전 비교 후 머지
- 동일 카드 동시 편집: 카드 단위 소프트 락(편집 중 표시) + last-write-wins + 토스트 알림
- 오프라인 편집: 로컬 큐에 쌓고 재연결 시 서버 버전과 리베이스
```
> 이 충돌 해소 설계가 "복잡한 비동기 상태"의 핵심 기술 포인트.

## 7. 성능 최적화 — 측정 기반 개선 계획

| 영역 | 측정 지표 | 개선 기법 | 기대 서사 |
|------|----------|----------|----------|
| 첫 진입(공유링크) | LCP, TTI | SSR + 코드 스플리팅 | "공유 일정 첫 로딩 N초 → M초" |
| 지도 | 마커 렌더 FPS, 노드 수 | 클러스터링 + 뷰포트 컬링 | "수천 마커에서 끊김 제거" |
| 장소 피드 | 스크롤 FPS, DOM 노드 | 가상화 + 이미지 lazy | "대량 리스트 부드러운 스크롤" |
| 협업 편집 | 리렌더 횟수 | 카드 메모이제이션 + 이벤트 throttle | "원격 변경당 리렌더 X→1" |
| 네트워크 | 요청 수/취소 | 검색 캐싱 + AbortController | "중복 검색 제거" |

## 8. 팀 역할 분담 (제안)

- **나 (FE 주력 + AI 서브)**: 에이전트 채팅 UI, **협업 보드 상태/충돌 해소**, 웹뷰 브릿지, 성능 최적화 / AI는 도구 스키마·오케스트레이션 흐름 설계
- FE 보조: 지도·장소 피드 UI, 디자인 시스템
- BE: 에이전트 백엔드(LLM 오케스트레이션), 실시간 협업 서버(권위 머지), 검색 도구 게이트웨이
- (인원에 따라) 데이터/AI: 동선 최적화·추천 로직

## 9. 마일스톤 (제안, 약 12주)

1. **W1–2 설계**: 상태 아키텍처·브릿지 인터페이스·협업 이벤트 프로토콜·에이전트 이벤트 프로토콜 확정
2. **W3–5 코어**: 에이전트 일정 생성 + 스트리밍 + 도구 시각화 (단일 사용자 보드)
3. **W6–7 실시간 협업**: WebSocket 동시 편집 + presence + 충돌 해소(1단계 락/LWW)
4. **W8 멀티모달/웹뷰**: 위치·카메라·푸시·딥링크 브릿지, 현장 플로우
5. **W9 지도/검색**: 마커 클러스터·동선·장소 피드 가상화
6. **W10 성능**: SSR·코드스플리팅·이벤트 throttle, 측정→개선 지표 확보
7. **W11 품질/도전**: 테스트·오프라인 재동기화·(도전) CRDT 순서 머지
8. **W12 마감**: 배포, 데모, 프로젝트 회고 정리

## 10. 핵심 기술 도전 과제

- **복잡한 비동기 상태 설계** → 실시간 동시 편집 충돌 해소 + 에이전트 상태머신 + 서버/클라 분리
- **다중 사용자 동기화** → presence·fractional index·오프라인 리베이스
- **성능 병목 측정·개선** → 지도 마커·가상화·이벤트 throttle before/after
- **SSR 적용** → 공유 링크 첫 진입·SEO
- **웹뷰 환경 문제 해결** → 브릿지 추상화, 지도 제스처/키보드/딥링크 이슈
- **AI UX 전달** → 에이전트 진행/도구 근거 시각화
- **AI 개발 도구(Claude Code/Cursor) 활용** → 개발 과정 기록

## 11. 리스크 & 대응

| 리스크 | 대응 |
|------|------|
| 동시 편집 난도 과다 | 1순위 "카드 락 + 낙관적 + LWW"로 동작, CRDT/OT는 도전 과제로 단계 분리 |
| 지도/항공 API 비용·쿼터 | 캐싱·목데이터 우선, 실호출은 데모 범위로 제한 |

| AI 비용/지연 | 도구 결과 캐싱, 스트리밍으로 체감 지연 완화, 모델 분기 |

## 부록 A. 동네비서와의 관계
- **공통**: 인앱 웹뷰 + Native Bridge + 멀티스텝 Agent + Agent 진행 시각화 + Zustand/TanStack 분리
- **차별**: 1:1 거래 채팅 → **N명 실시간 협업 편집(충돌 해소)**, 시세 검색 → **검색·지도·동선 + SSR**
- **선택 기준**: "동시 편집 동기화"라는 고난도 상태 문제를 깊게 파고 싶다면 트립크루.
